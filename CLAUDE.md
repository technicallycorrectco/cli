## Project

This is `@technicallycorrect/cli` — a TypeScript CLI (`tc`) for interacting with the Technically Correct API. The sibling directory `../technicallycorrect.io/` is the Phoenix/LiveView app that hosts the API this CLI calls.

## Commands

```bash
npm run build       # Compile TypeScript to dist/
npm run dev         # Run directly with tsx (no compile step)
npm run start       # Run compiled output
npm run generate    # Regenerate src/client/ from ../technicallycorrect.io/openapi.json
npm run lint        # Run ESLint
npm run format      # Run Prettier
npm run precommit   # format → build → lint — run before every commit
npm link            # Install tc globally for local testing
```

## Architecture

- **Entry point:** `src/index.ts` — registers all subcommands via Commander and calls `program.parse()`
- **Commands:** `tc config`, `tc r`, `tc t`, `tc i`, `tc d`, `tc p` — each maps to a subcommand module under `src/commands/`
- **Config:** Stored at `~/.technicallycorrect/cli/config.json`. A config module loads and merges values; all commands read from it for host, port, api-key, and default project slug.
- **Generated client:** `src/client/` is auto-generated by hey-api from `../technicallycorrect.io/openapi.json` via `npm run generate`. Never hand-edit files in this directory.
- **Services:** `src/services/` wraps the generated client with polling, auto-accept logic, and other CLI-specific behavior.
- **Output:** All responses are printed as JSON to stdout. Errors print to stderr and exit non-zero.

## API

The Technically Correct REST API is at `http(s)://<host>:<port>/api/v1/`. All endpoints require `Authorization: Bearer <token>`. URL structure: `/:org_slug/:project_slug/...`. The org slug is resolved from the API token — there is one org per token.

| Command | Method | Path |
|---|---|---|
| `tc r list` | GET | `/:org/:project/requirements` |
| `tc r <id>` | GET | `/:org/:project/requirements/:id` |
| `tc r create` | POST | `/:org/:project/requirements` |
| `tc r edit` | PATCH | `/:org/:project/requirements/:id` |
| `tc r accept` | POST | `/:org/:project/requirements/:id/accept` |
| `tc t list` | GET | `/:org/:project/tasks` |
| `tc t <id>` | GET | `/:org/:project/tasks/:id` |
| `tc t verify` | POST | `/:org/:project/tasks/:id/verify` |
| `tc i add` | POST | `/:org/:project/requirements/:id/implementations` |
| `tc d set` | PUT | `/:org/:project/requirements/:id/design` |
| `tc p list` | GET | `/:org/projects` |

## Task polling

`tc r create`, `tc r edit`, and `tc d set` are asynchronous — the API returns a task immediately and the CLI polls until `awaiting_review` or `complete`.

- **`tc r create`:** When `awaiting_review`, auto-accept and return the accepted requirement.
- **`tc r edit`:** When `awaiting_review` with no impact findings, auto-accept and return the accepted requirement. When there are impact findings, return the task result with affected requirements without accepting — the caller must review and call `tc r accept` or `tc t verify` explicitly.
- **`tc d set`:** When `awaiting_review` with no impact findings, auto-accept and return the updated requirement. When there are impact findings (affected implementations), return the task result without accepting.
